What is your favorite WordPress function? Why?

My favorite WordPress function of late is add_filter().

With it, a skilled developer can change nearly everything about how WordPress works. A few examples:

  • WordPress galleries offer an easy way for users to assemble a group of media. However, galleries’ default markup can be difficult to work with. I like to use add_filter to transform the markup into something that’s suitable for a slideshow.
  • WordPress extraneously wraps elements in paragraph tags, even images and videos. This isn’t very semantic. I like to use add_filter to hook onto the the_content to parse out the <p>s.
  • wp_title()’s default behavior is kind of annoying because it often doesn’t include the site’s title on many views. Adding it manually using a filtering function does the trick.

Name three good reasons to start a branch in a version control system.

  1. Working on a project’s next major version. Make a new branch and add your planned features. This way, your new features are isolated from the production branch. When you’re ready to launch, merge ‘em.
  2. Debugging something right now. What if – while fixing this bug – you inadvertently cause another new problem? Instead of working directly on the production branch, it’s safer to isolate your code in a new branch. Fix the bug and merge ‘em up.
  3. Trying out an experimental feature. Say you want to drop some new hotness on your project, but you’re not sure it will work. Start a new branch to play with that fire. Doesn’t work out? Toss your branch.

Why is it better to use CSS (rather than JavaScript, if possible) for a basic web page animation?

CSS animations and transitions run faster than their JavaScript counterparts.

A browser is able to execute CSS changes through the device’s GPU when relying on CSS3’s animation or transition. This is way faster than relying on JavaScript execution, which is regulated by the browser.

The difference is fairly negligible on a desktop computer with a modern browser; however, the issue becomes obvious on mobile devices, where JavaScript execution is an order of magnitude slower than desktops and JavaScript animations are choppy, slow and hold up the page.

Look at our portfolio and find something we've built that could be improved. Suggest how!

Equality House’s site looks great, reads great, and functions great; yet, there is a key area for improvement.

Let’s talk about performance, specifically stylesheets. Remember Yahoo!’s guidelines: minify files, use less HTTP requests, compress files, etc.

On the home page, the site loads five stylesheets. (There’s another one hiding on the product page.) One results from Typekit loading, so it can’t be eliminated. The other four, however, could be compressed into a single stylesheet. This would decrease file size (by minimizing plugins’ CSS) and reduce HTTP requests, making the page load and render faster. Here’s how:

  1. Use WordPress’s wp_dequeue_style in the theme’s functions.php to remove plugins’ stylesheets.
  2. Import all plugins’ stylesheets into app.css and minimize it.

This is more work on the developer: when a vendor updates his or her plugin, you have to manually recompile app.css. In the end, it pays off in performance. (One could add an action function to do this every time a plugin or theme is updated. Plugin idea, anyone?)

Another consideration is that none of the stylesheets have a Cache-Control header, most likely because WordPress likes to set query strings on assets. Consider setting app.css’s wp_enqueue_style’s $ver parameter to NULL so no query string is added.

Also, they’re not G-Zipped. Adding the appropriate rules to the global .ht_access should do the trick.
